# 抬腿逻辑提取文档

## 来源文件
- D:\downloads\PPO_episoid_2_1 (2).py
- D:\downloads\RobotRun2 (2).py

## 说明
本文档提取了两份文件中与抬腿阶段有关的核心逻辑，包括动作执行、离散门控、连续动作使用方式、奖励函数、成功判定、失败惩罚、回合结束、重置流程和训练日志。

## 一、整体结构
抬腿阶段主要分成两层：

1. 上层训练循环：负责调用智能体选动作、把动作送入环境、接收环境反馈、存储经验和学习。
2. 底层执行控制：负责把动作映射到关节、等待动作到位、读取 GPS 和触摸传感器、计算奖励和决定是否结束回合。

两份文件的侧重点不同：
- PPO_episoid_2_1 (2).py 更偏向训练编排和奖励设计，逻辑更完整，奖励更细。
- RobotRun2 (2).py 更偏向底层环境执行和回合判定，结构更直接。

## 二、动作选择与执行

### 1. 动作来源
抬腿阶段通过 tai_agent.choose_action 生成动作：
- 离散动作：用于门控，决定某个关节是否采用本步新动作。
- 连续动作：用于具体的关节增量或目标值。

### 2. 动作门控逻辑
在 PPO_episoid_2_1 (2).py 中，三路动作分别对应：
- 上腿关节
- 下腿关节
- 踝关节

执行时会先读取离散动作，再决定连续动作是否真正生效：
- 离散值为 1：使用本步新连续动作。
- 离散值为 0：保持上一步动作不变。

这样做的目的：
- 防止每一步都大幅改变关节。
- 让策略学习“哪些关节需要动，哪些关节保持”。
- 减少抖动和动作噪声。

### 3. 动作记录
训练脚本中会记录：
- 原始离散动作
- 原始连续动作
- 最终执行动作
- 动作对数概率
- 状态价值

同时还会把本步实际执行增量保存到 left_step_history 中，方便后续复盘或第三阶段复现。

## 三、状态构造

### 1. 抬腿特征状态
PPO_episoid_2_1 (2).py 中将抬腿状态构造成 40 维特征，主要包含：
- 双腿关键关节角度
- 这些关节的变化量
- 脚部相对目标的位置和距离
- IMU 特征
- 左脚触觉传感器值
- 上一步执行动作
- 上一步离散门控结果

### 2. 目的
这个状态设计的目标是让抬腿策略同时看到：
- 当前身体姿态
- 当前脚部是否接近目标
- 当前是否有接触反馈
- 上一步动作是否已经生效

这样奖励和动作就能形成闭环。

## 四、奖励逻辑

### 1. PPO_episoid_2_1 (2).py 中的奖励设计
该文件的奖励比较细，主要由以下几部分组成：

#### 1.1 距离惩罚
根据脚部当前点到目标点的距离 dist 给基础惩罚：
- 距离越远，惩罚越大。
- 用于约束整体方向。

#### 1.2 进度奖励
通过 prev_potential 记录上一步距离：
- 若本步比上一步更接近目标，则给正奖励。
- 若本步退步，则没有进度奖励，甚至会被其他项抵消。

#### 1.3 中心偏差惩罚
通过 center_error 计算横向偏差和纵向偏差：
- 对 dy 给予更大权重。
- 鼓励脚不仅接近目标，还要尽量踩在中心区域。

#### 1.4 中心进度奖励
如果当前 center_error 比上一步更小，则继续给进度奖励。
- 这让策略不仅追求“接近”，还追求“对准”。

#### 1.5 成功奖励
满足严格成功条件时给大奖励：
- goal == 1
- dist 小于 TAI_SUCCESS_DIST_THRESH

成功后还会按 closeness_ratio 再叠加额外奖励：
- 越接近中心，奖励越高。
- 这能进一步鼓励精确踩踏。

#### 1.6 偏踩惩罚
如果环境返回 goal == 1，但距离仍未达到阈值：
- 视为偏踩
- 取消成功
- 进入失败处理

#### 1.7 远离惩罚
如果脚部距离目标太远（dist > 0.25）：
- 额外扣分
- 让训练尽快结束无效回合

#### 1.8 偏踩补充惩罚
若 done == 1 且 goal == 0 且 dist 仍然不达标：
- 根据 miss_ratio 再加一次偏踩惩罚
- 越偏，惩罚越大

#### 1.9 动作惩罚
按执行动作幅度进行小惩罚：
- 防止动作过猛
- 促进平滑控制

#### 1.10 步数惩罚
每步都扣少量分：
- 鼓励更快完成抬腿
- 减少拖延行为

### 2. RobotRun2 (2).py 中的奖励设计
RobotRun2 (2).py 的奖励更接近底层执行逻辑：

#### 2.1 初始距离奖励
执行完动作后，先读取脚部 GPS，计算到目标点的距离：
- reward = -distance
- 距离越小，越有利

#### 2.2 成功条件奖励
当脚部触摸传感器触发，且 total_distance < success_dist_thresh 时：
- 认为踩踏成功
- reward = 50.0 + 80.0 * closeness_ratio
- goal = 1
- done = 1

#### 2.3 偏踩惩罚
当脚部触摸传感器触发，但 total_distance 没有达到阈值时：
- 认为踩到但偏位
- reward = -miss_penalty
- goal = 0
- done = 1

#### 2.4 未触发时的基础惩罚
如果没有脚部触摸：
- reward = -2.0
- 回合通常继续或进入后续逻辑

#### 2.5 碰撞惩罚
如果检测到 arm 或 leg 的碰撞触摸传感器触发：
- reward = -10.0
- 直接终止回合

## 五、成功判定

### 1. 严格成功判定
在 PPO_episoid_2_1 (2).py 中，严格成功条件是：
- goal == 1
- dist < TAI_SUCCESS_DIST_THRESH

只有两者同时满足才算成功。

### 2. 成功后的处理
成功时会：
- 设置 episode_success = True
- 输出成功日志
- 在 keep_pose_on_success 打开时保留当前姿态，准备进入第三阶段

### 3. 偏踩处理
如果 goal == 1 但距离不够：
- 认为不是严格成功
- 重新把 goal 置 0
- 终止回合
- 不进入成功保存逻辑

## 六、失败与终止逻辑

### 1. 终止条件
抬腿回合通常在以下场景结束：
- 触发成功
- 触发偏踩失败
- 步数达到 TAI_MAX_STEPS
- 碰撞或异常状态
- 传感器异常

### 2. 无效样本过滤
对于特别早结束且无有效接触的样本，会跳过经验存储：
- done == 1
- steps <= 2
- good != 1

这样避免把明显无效数据写进经验池。

### 3. 训练与推理模式
文件里支持 train_tai 开关：
- train_tai = True：会存储经验并调用 learn
- train_tai = False：只执行，不学习，不保存模型

## 七、回合重置与三阶段衔接

### 1. 正常回合结束后的重置
如果没有保留姿态：
- 重置左腿初始姿态
- 调用 robot_reset
- 调用 env.reset
- 等待稳定时间

### 2. 成功后保留姿态
如果 keep_pose_on_success 且 episode_success 为 True：
- 不立即完全重置
- 先检查左脚当前位置是否仍在目标邻域内
- 如果超出阈值，则取消三阶段训练并重置

### 3. 重置时的清理
重置流程中还会：
- 清空 imgs
- 删除临时张量引用
- 释放 GPU 缓存
- 等待若干仿真步稳定

## 八、日志与训练更新

### 1. 日志记录
抬腿阶段会记录：
- 回合编号
- 动作
- 对数概率
- 状态价值
- 累积奖励 return_all
- goal
- loss_discrete
- loss_continuous

### 2. 学习触发
在 done == 1 时，如果 train_tai 为真且满足训练条件：
- 调用 tai_agent.learn()
- 输出训练完成日志
- 写入 loss 到日志系统

### 3. 模型保存
满足保存条件时会保存模型：
- episode % 500 == 0
- done == 1
- episode > 100

保存内容包含：
- policy_tai
- optimizer_tai
- episode 编号

## 九、两份文件的核心差异

### 1. PPO_episoid_2_1 (2).py
特点：
- 更完整的训练逻辑
- 奖励设计更细
- 有进度奖励、中心奖励、偏踩惩罚、动作惩罚、步数惩罚
- 适合训练阶段的主逻辑参考

### 2. RobotRun2 (2).py
特点：
- 更底层、更直接
- 以触摸传感器和距离阈值判断是否成功

- 更像环境执行控制器

## 十、可直接复用的结论
如果后续要整理成统一版本，可以保留以下关键规则：
- 动作门控：离散值控制连续值是否生效
- 成功判定：触摸触发 + 距离阈值同时满足
- 失败判定：偏踩、远离、碰撞、超步数
- 奖励结构：距离惩罚 + 进度奖励 + 中心奖励 + 成功大奖励 + 偏踩惩罚 + 动作惩罚 + 步数惩罚
- 训练流程：先执行动作，再读取动作后 GPS 和触摸结果，再计算奖励和终止

## 十一、建议
如果你打算把这两份逻辑合并成一版主流程，建议按下面顺序保留：
1. 先做动作门控
2. 再执行关节动作
3. 动作后读取 GPS 和触摸
4. 根据距离和触摸判断 success / miss
5. 再计算 reward
6. 最后决定是否 learn 和 reset

这样训练时的状态、动作和奖励会更一致，也更容易调试。
