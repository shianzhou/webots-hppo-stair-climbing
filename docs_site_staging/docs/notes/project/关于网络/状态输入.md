# 三阶段状态输入说明

本文说明当前仓库中决策、抓取、踩踏三个 PPO 阶段分别使用什么状态输入。对应代码集中在：

- `E:\project_MultiAgent_h_change\python_scripts\PPO\state_inputs.py`
- `E:\project_MultiAgent_h_change\python_scripts\PPO\PPO_episoid_1.py`
- `E:\project_MultiAgent_h_change\python_scripts\PPO\hppo_01.py`
- `E:\project_MultiAgent_h_change\python_scripts\PPO\hppo.py`

## 总体结构

当前三阶段状态输入已经集中到 `state_inputs.py` 中：

| 阶段 | 构造函数 | 状态维度 | 主要用途 |
| --- | --- | --- | --- |
| 决策层 | `build_decision_state()` | 6 | 上层决策：判断进入抓取、踩踏或重新决策 |
| 抓取层 | `build_catch_state()` | 20 | 抓取 PPO：图像 + 机器人关节状态 |
| 踩踏层 | `build_tai_state()` | 40 | 踩踏 PPO：纯踩踏专用状态，无图像输入 |

三个维度常量是：

```python
DECISION_STATE_DIM = 6
CATCH_STATE_DIM = 20
TAI_STATE_DIM = 40
```

这些维度必须和 PPO 网络中的 `state_dim` 对齐，否则 checkpoint 或网络前向传播会维度不匹配。

## 1. 决策层状态输入

构造函数：

```python
build_decision_state(env, feature_dim=6)
```

返回：

```python
state, pressure_values, pressure_detected
```

### 输入内容

决策层使用 6 维手部压力传感器状态：

| 维度 | 传感器 |
| --- | --- |
| 0 | `grasp_L1` |
| 1 | `grasp_L1_1` |
| 2 | `grasp_L1_2` |
| 3 | `grasp_R1` |
| 4 | `grasp_R1_1` |
| 5 | `grasp_R1_2` |

状态构造逻辑：

```python
pressure_detected = any(pressure_values > 0.0)
```

注意：`pressure_detected` 是辅助流程/日志判断，不是额外状态维度。真正送入决策网络的是 6 维 `state`。

### 网络适配

决策层使用 `hppo.py` 中的 `HPPO`，初始化为：

```python
d_hppo(
    num_servos=1,
    node_num=19,
    state_dim=6,
    use_image_input=False,
)
```

所以决策层没有图像输入，图像分支由网络内部用零向量占位。

## 2. 抓取层状态输入

构造函数：

```python
build_catch_state(env, step, feature_dim=20)
```

返回：

```python
obs_img, obs_tensor, robot_state, obs, x_graph
```

### 输入内容

抓取层保持原有逻辑：图像 + 20 维机器人关节状态。

| 输入 | 内容 | 用途 |
| --- | --- | --- |
| `obs_img` | 当前图像，通常用于经验缓存 | `store_transition()` 的图像分支 |
| `obs_tensor` | 当前图像张量 | `choose_action()` 的图像分支 |
| `robot_state` | 20 维机器人关节状态 | 状态分支 |
| `obs=(obs_tensor, robot_state)` | 图像 + 状态组合 | 抓取策略输入 |
| `x_graph=robot_state` | 同一份 20 维关节状态 | 图/结构分支 |

20 维关节状态来自：

```python
env.get_robot_state()
```

并通过 `_fixed_state(..., 20)` 固定长度。如果少于 20 维会补 0，多于 20 维会截断。

### 为什么抓取仍保持 20 维

抓取阶段当前模型和历史 checkpoint 都基于 20 维关节状态。为了不破坏 catch 训练，抓取没有改成新的复杂状态输入。

抓取 agent 初始化为：

```python
hppo(num_servos=2, node_num=19, state_dim=20)
```

## 3. 踩踏层状态输入

构造函数：

```python
build_tai_state(
    env,
    robot_state,
    prev_robot_state,
    prev_exec_actions,
    prev_discrete_actions,
    feature_dim=40,
)
```

返回：

```python
tai_state
```

`tai_state` 是 `np.float32` 的 40 维向量。

### 输入内容总览

踩踏层状态不再直接使用原始 20 维 `robot_state`，而是构造专门面向左脚踩踏任务的状态：

| 维度范围 | 内容 | 含义 |
| --- | --- | --- |
| 0-5 | 腿部关键关节角 | 左右腿关键姿态 |
| 6-11 | 腿部关键关节变化量 | 当前关节状态 - 上一步关节状态 |
| 12-16 | 左脚到目标台阶的几何信息 | `foot_x, foot_y, dx, dy, dist` |
| 17-22 | IMU 特征 | 加速度计 + 陀螺仪归一化值 |
| 23-25 | 左脚触碰状态 | `foot_L1, foot_L2, foot_L3` |
| 26-28 | 上一步实际执行动作 | 三个左腿关节动作 |
| 29-31 | 上一步离散门控 | 三个关节是否采用新动作 |
| 32-39 | 预留 0 | 后续扩展踩踏特征 |

实际写入 32 个语义特征，最后通过 `_fixed_state(..., 40)` 补到 40 维。

### 关节角与变化量

腿部关键关节取：

```python
robot_state[10], robot_state[11], robot_state[12],
robot_state[13], robot_state[14], robot_state[15]
```

其中踩踏动作直接控制的是：

| 关节索引 | 关节名 |
| --- | --- |
| 11 | `LegUpperL` |
| 13 | `LegLowerL` |
| 15 | `AnkleL` |

变化量为：

```python
current_robot_state[i] - prev_robot_state[i]
```

它让策略知道关节是否正在运动、变化方向如何。

### 左脚到目标台阶的信息

左脚 GPS 来自：

```python
env.print_gps()[4]
```

目标台阶位置来自：

```python
gps_goal1
```

状态里写入：

```python
foot_x, foot_y, dx, dy, dist
```

其中：

```python
dx = goal_x - foot_x
dy = goal_y - foot_y
dist = sqrt(dx * dx + dy * dy)
```

这部分是踩踏状态增强的核心：策略可以直接看到左脚距离目标台阶还有多远、偏在哪个方向。

### IMU 特征

IMU 包含：

- 加速度计 `Accelerometer`
- 陀螺仪 `Gyro`

当前做了简单归一化：

```python
acc_feat = [(acc[0]-520)/100, (acc[1]-500)/100, (acc[2]-640)/120]
gyro_feat = [(gyro[0]-510)/20, (gyro[1]-510)/20, (gyro[2]-510)/20]
```

用途是让策略感知身体稳定性，避免只看脚部位置而忽略姿态失衡。

### 左脚触碰状态

踩踏状态读取三路左脚触碰：

| 维度 | 触碰传感器 |
| --- | --- |
| 23 | `foot_L1` |
| 24 | `foot_L2` |
| 25 | `foot_L3` |

`Webots_interfaces.py` 中已经补充了：

```python
'foot_L3': 'touch_foot_L3'
```

如果某个传感器读取失败，状态构造会回退为 `0.0`。

### 上一步动作与门控

踩踏策略输出包括：

- 三维连续动作：`LegUpperL / LegLowerL / AnkleL`
- 三维离散门控：是否使用本步新动作

状态里保存上一时刻：

```python
prev_exec_actions      # 上一步实际执行动作
prev_discrete_actions  # 上一步离散门控
```

这样策略知道上一拍到底执行了什么，避免每一步都像无记忆地重新选择动作。

### 网络适配

踩踏 agent 初始化为：

```python
hppo(num_servos=3, node_num=19, state_dim=40, use_image_input=False)
```

调用方式：

```python
tai_dict = tai_agent.choose_action(
    obs=tai_state,
    x_graph=tai_state,
)
```

经验缓存：

```python
state=[None, tai_state, tai_state]
next_state=[None, next_tai_state, next_tai_state]
```

这里的 `None` 表示踩踏阶段不再使用图像分支，网络内部用 100 维零向量占位。

旧 20 维踩踏 checkpoint 与 40 维新输入不兼容，会被跳过并从新网络重新训练。

## 4. 三者之间的主要区别

| 阶段 | 是否使用图像 | 状态维度 | 状态重点 |
| --- | --- | --- | --- |
| 决策 | 否 | 6 | 手部压力，用于判断抓取是否发生、是否可进入后续流程 |
| 抓取 | 是 | 20 | 原始关节状态，保持旧模型兼容 |
| 踩踏 | 否 | 40 | 左脚到目标台阶的距离、触碰、IMU、上一动作等任务特征 |

简化理解：

```text
决策：看手上有没有压力反馈。
抓取：看图像和全身关节状态来抓目标。
踩踏：不看图像，只看脚在哪里、离台阶多远、有没有触碰、身体稳不稳、上一拍做了什么。
```

## 5. 状态构造的安全策略

`state_inputs.py` 里所有状态都会经过清洗：

```python
validate_and_clean_data()
_fixed_state()
```

作用是：

- 替换 NaN / Inf。
- 固定输入维度。
- 少了补 0，多了截断。
- 传感器读取失败时使用 0。

这保证了 PPO 网络不会因为某个传感器临时异常而收到 NaN 或长度不对的输入。

## 6. 当前需要注意的点

1. 抓取和踩踏都使用 `hppo_01.py`，但状态维度不同：抓取 20，踩踏 40。
2. 决策使用 `hppo.py`，且 `use_image_input=False`。
3. 踩踏使用 `hppo_01.py`，但同样设置 `use_image_input=False`，不再读取图像。
4. 如果以后修改 `TAI_STATE_DIM`，必须同步修改踩踏 agent 初始化和 checkpoint 兼容策略。
5. 如果新增传感器输入，优先加到 `state_inputs.py`，不要直接在 `PPO_episoid_1.py` 主流程里拼状态。
6. 踩踏状态中 32-39 目前是预留 0，可以用于后续加入右脚位置、身体高度、目标法向等特征。
