# RobotRun0 决策层：总流程与动作逻辑

对应代码：`E:\project_MultiAgent_h_change\python_scripts\PPO\RobotRun0.py`

## 一句话定位

`RobotRun0` 是上层决策层的包装器。它不直接控制 Webots 电机，而是根据决策模型输出的 `decision=0/1` 和当前 `catch_success` 状态，决定本轮走抓取、踩踏，还是重新决策。

## 输入从哪里来

决策层状态由 `state_inputs.py` 中的 `build_decision_state(env)` 构造：

```text
6 维手部压力状态 -> decision_agent.choose_action() -> decision
```

6 维顺序为：

| 维度 | 传感器 |
| --- | --- |
| 0 | `grasp_L1` |
| 1 | `grasp_L1_1` |
| 2 | `grasp_L1_2` |
| 3 | `grasp_R1` |
| 4 | `grasp_R1_1` |
| 5 | `grasp_R1_2` |

这 6 维压力状态会传给：

```python
decision_dict = self.decision_agent.choose_action(obs=decision_state_input)
decision = int(decision_dict["discrete_action"][0])
```

## 决策动作如何映射

`decision` 只有两个主要动作：

| decision | 含义 |
| --- | --- |
| `0` | 选择抓取阶段 catch |
| `1` | 选择踩踏阶段 tai |

但最终路线不只看 `decision`，还要看当前是否已经抓取成功，即 `catch_success`。

## 路由逻辑

`judge_route(decision, catch_success)` 的逻辑是：

| 条件 | 返回路线 | 原因 |
| --- | --- | --- |
| `decision == 0` 且 `catch_success == False` | `catch` | 还没抓到，需要抓取 |
| `decision == 0` 且 `catch_success == True` | `re_decide` | 已经抓到了，不该重复抓取 |
| `decision == 1` 且 `catch_success == True` | `tai` | 已抓取成功，可以踩踏 |
| `decision == 1` 且 `catch_success == False` | `re_decide` | 还没抓取，不能踩踏 |

所以 `RobotRun0` 的核心不是“执行动作”，而是“根据业务状态过滤决策动作”。

## 总执行流程

```text
build_decision_state(env)
  -> RobotRun0.choose_action()
  -> decision=0/1
  -> RobotRun0.judge_route(decision, catch_success)
  -> route = catch / tai / re_decide
  -> 主流程根据 route 调用 run_catch_stage 或 run_tai_stage
  -> RobotRun0.finalize() 计算决策奖励并训练决策层
```

## 奖励逻辑

`_compute_reward()` 按决策是否合理给奖励：

| 情况 | 奖励 |
| --- | --- |
| 未抓取时选择抓取，且抓取成功 | `+5.0` |
| 未抓取时选择抓取，但抓取失败 | `-8.0` |
| 已抓取时又选择抓取 | `-15.0` |
| 已抓取时选择踩踏 | `+10.0` |
| 未抓取时选择踩踏 | `-10.0` |

这个奖励训练的是“什么时候应该进入哪个子阶段”，不是训练抓取或踩踏动作本身。

## 容易混淆的点

1. `RobotRun0` 不控制电机。
2. `decision=1` 不一定会进入踩踏，必须 `catch_success=True`。
3. `pressure_detected` 只是辅助打印/流程信息，不是额外网络维度。
4. `finalize()` 里才会把决策经验写入 buffer 并触发决策模型学习。
